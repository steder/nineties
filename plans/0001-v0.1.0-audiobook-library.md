# Nineties — A Curated, Low-Tech Media Experience for Kids

## Context

This project recreates a pre-streaming relationship to media — finite, scheduled, deliberate, with friction — for the user's kids (ages 9, 5, and 4 months), who are presently shaped by Audible, Alexa, and binge streaming. The user (an elder millennial, heavy Plex user) wants something **libre / open source**, family-first but publicly shareable, that gives **parents fine-grained curation control** while **giving kids real agency within those bounds**.

The product is *not* another media server. The OSS media server problem is solved (Jellyfin, AudioBookShelf, Navidrome, ErsatzTV are all excellent). The product is the **curation policy + library/scarcity semantics + kid-facing front door** wrapped around those servers, plus the **escape hatch** for parents.

The acknowledged irony: we are using Docker, Postgres, Django, an Alexa cloud skill, and several modern OSS servers to *simulate* the experience of a library card, a CD shelf, and a TV with a published schedule. The 90s ideal is the **shape of the experience for the child**, not the substrate. The README will say this out loud.

---

## Guiding principles

1. **Finite by default.** Catalogs are bounded subsets, not "everything we own." Loans expire. Today's TV is *today's* TV.
2. **Tags over ratings.** MPAA/ESRB-style ratings are too coarse and morally pre-loaded (curse words ≠ violence). Items are tagged on multiple value axes; families define which axes matter and at what thresholds.
3. **Per-kid policy.** A 9yo and a 5yo cannot share a catalog. Policy is keyed on kid identity, not family. This is a v1 hard requirement, not a v1.1 stretch.
4. **Boredom is a feature.** "Nothing on right now" is a legitimate state. The UI should let it be quiet.
5. **Escape hatch, not bypass.** Parents get to the full underlying servers with a PIN — friction is a feature, but a wall isn't.
6. **Own your data, own your stack.** Self-hosted; no telemetry; no kid-data leaves the house except where Alexa requires it (and only the minimum). Built-in path to remove Alexa entirely later.
7. **Pluggable boundaries.** The two places we'll regret hardcoding are the **voice adapter** (Alexa today, local STT/LLM/TTS tomorrow) and the **media backend** (AudioBookShelf today, maybe Plex/Jellyfin/custom later). Both are abstracted from day one.
8. **License: AGPL-3.0.** Strong copyleft suits "libre" intent for a networked service.
9. **Agentic-friendly by construction.** Fast, deterministic test/lint/type loops; every meaningful feature has a property test or fixture-driven scenario. AI agents should be able to build and verify most work without human-in-the-loop except at design and last-mile testing.

## v1 scope — Audiobook library + Alexa front door

A working system on the user's home network where:

- The user has a curated catalog of audiobooks ingested by **AudioBookShelf**.
- A **Librarian** service on top exposes a curated subset to each kid, enforces a loan model (e.g., 2 active loans, 1-week expiry), and tracks listening progress.
- The 9yo and 5yo each have their own policy; the 4mo is a configured-but-inactive profile.
- A kid says **"Alexa, ask Story Time to read me my book"** on a bedside Echo and resumes their current loan. They can browse what's available, check one out, and return it by voice.
- A parent has a Django-admin-powered web UI for ingesting books, tagging, defining per-kid policy, viewing listening history, and overriding (PIN) to grant any book on demand.

No video, no music, no tablet client, no bedside Pi in v1. Those are explicit Phase 2/3.

│                  │ HTTPS│  - Django admin            │ HTTP │  (Docker, OSS)     │
│  Voice adapter:  │      │  - voice adapter (Alexa)   │      │  - files, metadata │

└──────────────────┘      │  - policy / loans / tags   │      └────────────────────┘

**Librarian** is the only new service we build. AudioBookShelf is the source of truth for *files and basic metadata*. Librarian is the source of truth for *who can borrow what, when, and what they've listened to*. Postgres holds Librarian's data, including (later) embeddings for fuzzy lookup.

The Alexa skill backend is **a Django view inside Librarian** — same process, same DB, just a different URL handling Alexa request JSON. Self-hosted (via Cloudflare Tunnel) is the default; a Lambda-deployable variant is a future option since the voice adapter is pluggable.

## Components

### 1. AudioBookShelf (existing OSS, Docker)

Chosen after a brief comparison:

| Option                        | Pros                                              | Cons                                              |
|-------------------------------|---------------------------------------------------|---------------------------------------------------|
| **AudioBookShelf** (pick)     | Active, audio-first, mobile + web clients, OPDS, signed streaming URLs, Home Assistant integration via Wyoming emerging | Yet another service to run |
| Booksonic / -air              | Subsonic-compatible (rich client ecosystem)       | Less audiobook-aware, slower development pace     |
| Plex audiobooks               | Already running it                                 | Audiobook UX is a second-class citizen            |
| Roll our own (just file serving) | One fewer container                            | Reimplements ripping/metadata/clients for no gain |

AudioBookShelf wins clearly for v1. We isolate it behind an `MediaBackend` interface in Librarian so switching later is bounded.

### 2. Librarian (new — this repo)

- **Language**: Python 3.14 (current stable; PEP 779 free-threading available, PEP 750 t-strings, deferred annotation evaluation).
- **Framework**: **Django 5.x**. Rationale: the parent UI is the heaviest UX in v1 (ingest queue, tagging, per-kid policy editor, history) and Django admin solves ~70% of that for free. Django ORM with native Postgres/pgvector support, mature migrations, ASGI for the Alexa endpoint, built-in auth for parents. `django-ninja` available if we want a typed JSON API surface later.
- **Storage**: **PostgreSQL 18.x** with `pgvector` extension and native `tsvector` full-text search (GIN indexes).
- **Background jobs**: `django-q2` or `huey` (single-process Redis-free options) for v1. Celery only if we outgrow it.
- **Auth**:
  - Kids → no password; identified by voice-adapter device ID (Alexa `deviceId`) or web-client cookie (kid mode is PIN-less by design).
  - Parents → Django auth with PIN escalation for sensitive actions (override, policy edits).

Library subpackages:

- `librarian.policy` — eligibility / tagging logic (pure functions, heavily tested)
- `librarian.loans` — checkout / return / expiry
- `librarian.media.abs` — the `MediaBackend` interface implementation for AudioBookShelf
- `librarian.voice.alexa` — the `VoiceAdapter` implementation for Alexa
- `librarian.jobs` — periodic sync, expiry sweep, embedding refresh

### 3. Alexa custom skill (new — this repo)

- **Skill model**: Custom skill, **developer mode** (private to user's Amazon account — no store submission, no certification, free).
- **Invocation name**: e.g., `"story time"` (configurable; user picks during setup).
- **Intents** (v1):
  - `ResumeListeningIntent` — "read me my book", "keep going"
  - `BrowseAvailableIntent` — "what books can I check out?"
  - `CheckoutBookIntent` — "check out {Book}"
  - `ReturnBookIntent` — "I'm done with {Book}"
  - `GoodnightIntent` — "good night Story Time"
  - `AMAZON.HelpIntent`, `AMAZON.StopIntent`, `AMAZON.CancelIntent` (required)
- **Audio playback**: Alexa `AudioPlayer.Play` directive with a signed streaming URL produced by Librarian (proxying AudioBookShelf). Long-form audio is fully supported including pause/resume across sessions.
- **Device → kid mapping**: simple config table mapping Alexa `deviceId` → kid identity. Each kid's bedside Echo "is" them. (Account linking is overkill for v1.)
- **Hosting**: Self-hosted via Cloudflare Tunnel pointed at Librarian. The voice adapter abstraction allows a future AWS Lambda variant (the same handler running in a Lambda-shaped wrapper) for parents who can't expose a home server.

## Data model (Librarian — v1)

```
Kid          (id, name, birthdate, alexa_device_ids[], active, ...)
Tag          (id, key, value, axis)            # axis: language|violence|theme|scariness|age_floor|...
Item         (id, abs_id, title, author, narrator, duration, cover, tags[], embedding vector(768))
                                               # abs_id = AudioBookShelf id
                                               # embedding for fuzzy lookup, later
Policy       (id, kid_id, axis, max_value, mode)   # mode: allow|deny|require_review
Loan         (id, kid_id, item_id, checked_out_at, expires_at, returned_at?, progress_seconds)
ListenEvent  (id, loan_id, occurred_at, position_seconds, kind)
                                               # kind: start|pause|resume|complete
OverrideLog  (id, parent_id, kid_id, item_id, granted_at, reason)
```

Postgres-specific niceties:

- `Item.title`, `author`, `narrator` indexed via a `tsvector` GIN index for fast text search.
- `Item.embedding` is a `pgvector` column for future fuzzy lookup ("the dragon book", "the one Mom and I started").
- `Tag` is normalized; `Item.tags` is a join, not an array, so we can index and query both directions.

Key invariants:

- A kid has at most `family.max_active_loans` non-returned loans.
- A loan with `expires_at < now` and no `returned_at` is automatically returned by a periodic job (progress preserved for next checkout — a real library doesn't forget your bookmark even if you returned it).
- An item is **eligible** for a kid iff: for every policy where `mode in (deny, require_review)`, the item has no matching tags; and for every `mode=allow`, the item has a matching tag if required.

## Curation / policy model (the heart of the project)

**Tagging is multi-axis, not a single number.** Example axes:

| Axis        | Example values                                           |
|-------------|----------------------------------------------------------|
| language    | mild, moderate, explicit                                 |
| violence    | none, cartoon, realistic, graphic                        |
| themes      | death, divorce, bullying, romance, religion, war         |
| scariness   | none, suspenseful, scary, horror                         |
| age-floor   | 4, 6, 8, 10, 12, 14                                      |

Each family defines what each axis means *to them* and at what threshold an item is blocked or flagged for review. Defaults ship as a starting point; they're not normative.

**Tag sources** (in priority order):

1. Parent override / manual tag on an item (final word).
2. Crowdsourced community tags (future, opt-in, federated — *not* in v1).
3. Metadata from AudioBookShelf / Open Library / Common Sense Media import, mapped via heuristics.
4. LLM-assisted tag suggestion at ingest time (later, opt-in — local LLM via Ollama).

**New items default to "needs review"** — un-tagged items don't appear in any kid's catalog until a parent has tagged them. This is the friction that makes the catalog deliberate.

---

## Library / loan mechanics

- **Max active loans per kid**: family-configurable, default 2.
- **Loan duration**: default 7 days. Configurable per family, per item, or per kid.
- **Return**: explicit ("I'm done") or automatic on expiry.
- **Cooldown**: same item can't be re-borrowed for N days after return (default 7). Promotes variety. Overridable by parent.
- **"On the shelf today"**: a daily-rotated subset of the eligible catalog is featured in the kid's browse list. The full eligible catalog is also searchable, but the *front page* is small and curated by the system — this is the "new arrivals shelf" feel.
- **Returns preserve progress.** Resume from where you stopped on the next checkout.

---

## Parent override (escape hatch)

- Django admin has a "grant access" action per item and per kid.
- Voice override path: `"Alexa, ask Story Time for parent mode"` → prompts for 4-digit PIN spoken as digits → opens a 5-minute window where any catalog item can be checked out, including un-reviewed items.
- All overrides logged with reason field.
- Hard parent escape hatch: AudioBookShelf itself remains reachable on the LAN at `audiobookshelf.local` with parent credentials. Bypasses Librarian entirely. This is documented as intentional.

## Future vision — removing Alexa (Phase 4)

The voice adapter abstraction is *the* thing that makes this future-compatible. The kid-facing experience stays the same; only the implementation changes:

- **Wake word**: openWakeWord (OSS) — custom phrase like "Storyteller"
- **STT**: whisper.cpp / faster-whisper, local
- **Intent + slot extraction**: small local LLM via Ollama (Llama 3.x / Qwen / Phi) — much more forgiving of kid speech and fuzzy book titles than Alexa's slot model
- **Lookup**: Postgres FTS + pgvector for "the one about the dragon Dad and I started" — *this is why the storage choice matters now*
- **TTS**: Piper (OSS, runs on Pi)
- **Orchestrator**: Wyoming protocol (Home Assistant) — emerging open standard for piping these together
- **Hardware**: Raspberry Pi 5 or similar with mic array + speaker, replacing the bedside Echo

The point: choosing Postgres + pgvector and a `VoiceAdapter` interface *now* makes this a refactor later, not a rewrite.

## Testing strategy (agentic-friendly by design)

The goal is a feedback loop tight and trustworthy enough that an AI agent can implement → run tests → fix → re-run without a human stepping in until the feature is done. That means tests must be **fast**, **deterministic**, **isolated**, and **exercise the things that matter for safety** (policy/eligibility).

### Test layers

| Layer                | Tool                                      | What it covers                                                                       |
|----------------------|-------------------------------------------|--------------------------------------------------------------------------------------|
| Unit                 | pytest                                    | Pure functions: policy eligibility, loan-lifecycle helpers, tag matching             |
| Property             | Hypothesis                                | Invariants on the policy engine — e.g., "no item with `age_floor > kid.age` ever appears in browse for that kid" |
| Integration          | pytest-django + testcontainers            | Real Postgres + pgvector; ABS sync against a stubbed ABS server (or a real one in compose) |
| Snapshot             | syrupy                                    | Alexa response JSON shapes; admin templates                                          |
| Voice E2E (Alexa)    | pytest + recorded Alexa request fixtures  | Replay real Alexa request JSON through the full Django stack against a seeded DB     |
| Browser E2E          | Playwright (`pytest-playwright`)          | Parent UI flows: ingest queue → tag → policy editor → override; built-in axe-core for accessibility |
| User-journey (BDD)   | pytest-bdd (Gherkin `.feature` files)     | High-level scenarios that match the v0.1.0 verification checklist 1:1; AI agents implement against these specs |
| Smoke                | `just smoke`                              | Full happy path: ingest → tag → checkout → playback URL signed → progress recorded   |

**Where BDD lives**: `tests/features/*.feature` for user-journey scenarios only (e.g., `kid_checks_out_book.feature`, `parent_overrides_block.feature`). Step definitions in `tests/steps/`. Plain pytest is preferred everywhere else — BDD's overhead only pays off when scenarios are the load-bearing spec.

### Fixture infrastructure

- **`fixtures/catalog/`** — a small "fake but realistic" catalog (silent .m4b files with metadata) committed to the repo. Lets every test/CI run exercise ABS sync end-to-end without external dependencies.
- **factory-boy** factories for `Kid`, `Tag`, `Item`, `Policy`, `Loan` with sensible defaults.
- **Recorded Alexa requests** in `fixtures/alexa/` — captured from real device interactions, replayable in tests.
- **Gherkin feature files** in `tests/features/` — also serve as living documentation of what the system promises to do.

### just recipes (the agent's hands)

`just` is a cross-platform task runner (single static binary, works on Windows / macOS / Linux). Recipe bodies prefer `uv run`, `docker compose`, and `gh` over raw shell builtins so they work identically across OSes; the rare shell-specific bit (file globbing, etc.) is wrapped in a small Python script. Windows users are recommended to use WSL2 for the Docker side but `just` itself runs natively.

- `just test` — all unit + property + integration tests; under 60s target
- `just e2e` — Alexa request replay + Playwright browser E2E against a spun-up compose stack
- `just bdd` — Gherkin scenarios only (subset of e2e, useful in agent loops)
- `just smoke` — single happy-path run (~10s), the agent's "did I break anything obvious?" check before claiming a task is done
- `just lint` — ruff
- `just typecheck` — pyright with django-stubs
- `just dev` — bring up the dev compose stack
- `just fmt` — ruff-format + just-format
- `just ci` — what CI runs; should match what an agent runs locally
- `just install-hooks` — install pre-commit hooks (ruff, ruff-format on staged files)

### Quality gates

The `REVIEW.md` rubric is enforced both by code review and by CI: any change to `librarian/policy/` requires a corresponding property test; any change to `voice/alexa/` requires a snapshot test against recorded requests; new migrations require a migration plan in the PR description.

## Phasing (post-v1)

- **Phase 2 — Music (Navidrome)**: Same Librarian patterns, second `MediaBackend`. "Today's mix" rotates. Kid-facing web client.
- **Phase 3 — Bedside Pi / tablet client**: Simple kiosk web app (already served by Librarian) for kids who can't talk to Alexa, or when Alexa is asleep. Buttons not feeds.
- **Phase 4 — Local voice stack** (remove Alexa): second `VoiceAdapter` implementation; Wyoming-protocol-based. The infrastructure for this is the embeddings + FTS we set up in v1.

- **Phase 6 — Federation of community tags** (opt-in): OSS-flavored answer to "I don't want to tag 4,000 audiobooks alone." Not before v1 has real-world miles.

## Verification — how we know v1 works

1. **Ingest**: parent uploads 5+ audiobooks to AudioBookShelf; Librarian sync job lists them as `needs_review` in Django admin.
2. **Tag**: parent tags each item across axes via Django admin; sets per-kid policies for the 9yo and 5yo.
3. **Browse via Alexa**: 5yo says invocation phrase on their bedside Echo; Alexa lists 3 eligible items (filtered by their policy). 9yo on a different Echo sees a different list.
4. **Checkout**: kid checks one out; Librarian records `Loan` row.
5. **Listen**: Alexa plays the audio; pause/resume across sessions persists progress.
6. **Listen via web**: parent can resume the same book on the parent web UI for QA.
7. **Expiry**: forcing a loan's `expires_at` to past triggers auto-return; progress preserved.
8. **Override**: parent grants a `needs_review` item to a kid via PIN; entry shows in override log.
9. **Policy block**: a tagged-too-mature item does not appear in browse for the 5yo, *does* appear for the 9yo.
10. **End-to-end Alexa flow** tested on the actual Echo device, not just the Alexa developer console simulator.
